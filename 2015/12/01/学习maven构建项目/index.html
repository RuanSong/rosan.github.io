学习Maven构建项目

  • java
  • maven
  • 后端
  • 学习笔记

Maven是一个项目管理以及自动构建工具，Maven使用称为项目对象模型(Project Object Mode,POM)来配置，基于maven构建项目可以高效的管理第三方的jar包，可以摆脱基于eclipse开发项目手动复制大量jar包以及jar包冲突等尴尬项目开发情景，这里主要记录学习maven一些步骤，方便日后查阅。

Maven的安装

Maven是由Apache软件基金会所提供的，因此我们获取maven要到apache的maven官网上面去获取maven的安装包。
下载完成之后，我们需要将maven加入到系统环境变量当中，并将maven目录下的bin目录映射到系统环境变量的path下，这样我们就可以全局使用maven了。

Maven的项目目录结构

|-src/ (maven项目根目录)
|—main/ (maven项目主体文件目录根)
|——java (maven项目源代码目录)
|———package (maven项目源代码包结构)
|——resources (maven项目资源目录)
|—test/ (maven项目测试目录根)
|——java (maven项目测试代码目录)
|———package (maven项目测试代码包结构)
|——resources (maven项目测试资源目录)
|-pom.xml (maven的POM文件)
|-target/ (maven输出的目录根)
|—classes (maven项目主体输出目录)
|—test-classes (maven项目测试输出目录)

maven中的坐标和仓库

坐标

groupId、artifactId、version三个名称是项目的坐标，唯一标识项目

仓库

1.本地仓库
本地仓库是存在到本地电脑的第三方jar，通过mvn install命令安装的maven项目也会被装到本地仓库中，本地仓库的路径通过maven配置文件setting.xml中的localRepository去指定。
如果项目中引用到某一个坐标，maven会首先到远程仓库里面找，如果找不到，maven会到远程仓库中下载，并且把下载的版本保存在本地仓库当中。
2.远程仓库
远程仓库存放着一些开源软件所提交的maven项目托管，maven的世界中央仓库地址:http://search.maven.org/。 远程仓库的地址可以在maven配置文件**setting.xml**中设置mirrors标签可以指定远程仓库镜像。 3.镜像仓库世界中央仓库的备份，有的时候国内访问国外或是访问不了，或是网速慢，国内也有一些maven远程仓库去定时更新maven中央仓库的项目，我们可以通过镜像仓库去下载第三方的maven项目`。

国内的镜像仓库:htt·p://mvnrepository.com/

Maven的生命周期和插件

一个项目的完整流程
maven完成了一个项目完整生命周期的管理

clean 、 compile、test、package、install、deploy
清理编译、测试、打包、集成测试、验证、部署

maven生命周期
clean 清理项目
pre-clean 执行清理前的工作
clean 清理上一次构件生成的所有文件
post-clean 执行清理后的文件
default 构建项目(最核心)
compile test package install
site 生成项目站点
pre-site 在生成项目站点前要完成的工作
site 生成项目的站点文档
post-site 在生成项目站点后要完成的工作
site-deploy 发布生成的站点的到服务器上

pom.xml 解析

pom.xml里面是日常开发当中用的最多的，我们通过这个pom文件我们可以指定项目所需的依赖范围，项目的坐标等关键信息。

依赖的范围

在项目依赖当中，有些依赖并不是我们在项目的整个周期中都需要的，因此我们可以通过scope指定项目所依赖的范围，通过scope指定之后，这个依赖会在特定的阶段加入特定的classpath路径当中。

在上述依赖例子中我们用到junit，我们只需要在测试的时候用到，因此我们设定junit的scope为test
scope可设置的值

test:测试范围内的classpath
complie(默认级别):编译、测试、运行 classpath都有效
provided:在编译和测试时的classpath有效，最后运行的classpath时候无效
runtime:测试和运行的classpath中有效
test:测试的classpath
system:与provided一样，移植性较差
import:只用在dependencyManagement中，表示是其他项目中的模块

依赖的传递

maven的依赖是具有传递性的，例如现在有三个模块ModuleA,ModuleB,ModuleC,其中ModuleA引用了Junit，ModuleB引用了ModuleA,当ModuleC依赖ModuleB的时候就会默认将ModuleA所应用的Junit给引用进来。

这种依赖的关系可以通过exclusions来排除

通过exclusions之后ModuleC模块在依赖ModuleB的时候就不会引用junit了

依赖的冲突

当ModuleA依赖commons-io的2.0版本，ModuleB依赖commons-io的2.4版本的同时，ModuleC同时依赖ModuleA和ModuleB，这个时候ModuleC的依赖就会产生冲突，maven在产生冲突的时候有两种优先方式来处理。
1.短路优先

2.先声明先优先
如果路径相同，则看POM文件中谁先声明，就先解析谁

聚合和继承

当几个Module的相互引用关系需要被调用的时候，通常需要分别依赖其他的几个模块，或是多个模块同时引用同一模块的时候，我们往往要在POM文件中写入冗余的依赖关系，Maven提供了聚合和继承两种方式来处理这种冗余。

无论是聚合和继承，都需要将<packaging>中的属性设置为pom,通过maven的install命令将聚合或是继承之后的坐标安装至本地仓库当中，这样就可以对外提供一次的依赖达到多次依赖的效果。

将maven共用的一些属性设置在properties文件中，在依赖的时候可以根据${}表达式的方式来获取全局的执行配置，达到全局配置的目的。
第二步，子pom设置parent指定父pom

子pom在引入依赖的时候就可以通过groupId和artifactId的方式来引入父pom中所声明的junit依赖了，而不要声明版本号。

Maven的插件

maven通过插件的方式可以指定在maven的生命周期阶段加入扩展的功能，这里以maven的官方插件source来作为maven插件的演示maven插件。source插件是在项目的打包过程中将maven的主体代码生成源文件的resource包。
maven插件是声明在pom文件的bulid标签下的,在配置中我们先声明maven的插件为source，这里引入source的2.4插件。

然后我们通过execution声明插件的运行时机为package

最后我们通过goals为该插件指定在运行的时候所执行的目标为jar-no-fork,也就是在打包的时候只把当前项目的源文件给导出为源文件jar包。

Maven构建Web项目

Maven构建Web项目只是在maven的main目录下新建了web-app目录，该目录下存放了Web相关的目录信息，构建Web项目时候我们要在POM文件中加入Servlet相关的POM依赖。这里的依赖范围要设置为provided,指明是在编译和**测试阶段才引入这个依赖。

在声明Web项目的时候要将POM的packaging设置为war,这样在打包的时候就会产生Web容器所识别的war包。

另外我们要以插件的方式引入Web容器。

这样我们通过mvn jetty:run命令就可以以jetty的方式来启动我们的jetty容器了。但是这样比较麻烦，我们可以指定jetty插件在项目打包完毕之后就启动，我们可以给jetty插件指定打包完成之后需要执行的目标。

这样我们只需要运行mvn package命令就将我们的生成的war包放到Web容器下，并启动Web容器。
如果我们要换其他的Web容器，如tomcat，我们只需将jetty插件换成tomcat件即可，不需做其他改动。

总结

通过以上的总结，我们可以看到，在引入maven管理之后，整个项目的的依赖关系通过POM文件可以看得一清二楚，依赖的冲突、依赖的范围、依赖的聚合和继承可以让我们更高效的组织POM文件，让我们从jar的复制，解决jar包冲突，右键编译，运行，生成war包、部署等杂乱无章的事情中解脱出来了，从而真正的关心我们开发的模块代码该怎么写。在团队协作方面，每个人负责一个项目模块，用maven进行管理并最后组织起来不失为一种提升效率的方式，不得不说，maven是项目开发的利器。
