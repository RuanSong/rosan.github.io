学习Maven构建项目

Maven是一个项目管理以及自动构建工具，Maven使用称为项目对象模型(Project Object Mode,POM)来配置，基于maven构建项目可以高效的管理第三方的jar包，可以摆脱基于eclipse开发项目手动复制大量jar包以及jar包冲突等尴尬项目开发情景，这里主要记录学习maven一些步骤，方便日后查阅。

Maven的安装

Maven是由Apache软件基金会所提供的，因此我们获取maven要到apache的maven官网上面去获取maven的安装包。
下载完成之后，我们需要将maven加入到系统环境变量当中，并将maven目录下的bin目录映射到系统环境变量的path下，这样我们就可以全局使用maven了。

maven中的坐标和仓库

坐标

groupId、artifactId、version三个名称是项目的坐标，唯一标识项目

仓库

1.本地仓库
本地仓库是存在到本地电脑的第三方jar，通过mvn install命令安装的maven项目也会被装到本地仓库中，本地仓库的路径通过maven配置文件setting.xml中的localRepository去指定。
如果项目中引用到某一个坐标，maven会首先到远程仓库里面找，如果找不到，maven会到远程仓库中下载，并且把下载的版本保存在本地仓库当中。
2.远程仓库
远程仓库存放着一些开源软件所提交的maven项目托管，maven的世界中央仓库地址:http://search.maven.org/。 远程仓库的地址可以在maven配置文件**setting.xml**中设置mirrors标签可以指定远程仓库镜像。 3.镜像仓库世界中央仓库的备份，有的时候国内访问国外或是访问不了，或是网速慢，国内也有一些maven远程仓库去定时更新maven中央仓库的项目，我们可以通过镜像仓库去下载第三方的maven项目`。

pom.xml 解析

pom.xml里面是日常开发当中用的最多的，我们通过这个pom文件我们可以指定项目所需的依赖范围，项目的坐标等关键信息。

依赖的范围

在项目依赖当中，有些依赖并不是我们在项目的整个周期中都需要的，因此我们可以通过scope指定项目所依赖的范围，通过scope指定之后，这个依赖会在特定的阶段加入特定的classpath路径当中。

在上述依赖例子中我们用到junit，我们只需要在测试的时候用到，因此我们设定junit的scope为test
scope可设置的值

test:测试范围内的classpath
complie(默认级别):编译、测试、运行 classpath都有效

system:与provided一样，移植性较差
import:只用在dependencyManagement中，表示是其他项目中的模块

聚合和继承

当几个Module的相互引用关系需要被调用的时候，通常需要分别依赖其他的几个模块，或是多个模块同时引用同一模块的时候，我们往往要在POM文件中写入冗余的依赖关系，Maven提供了聚合和继承两种方式来处理这种冗余。

无论是聚合和继承，都需要将<packaging>中的属性设置为pom,通过maven的install命令将聚合或是继承之后的坐标安装至本地仓库当中，这样就可以对外提供一次的依赖达到多次依赖的效果。

Maven构建Web项目

Maven构建Web项目只是在maven的main目录下新建了web-app目录，该目录下存放了Web相关的目录信息，构建Web项目时候我们要在POM文件中加入Servlet相关的POM依赖。这里的依赖范围要设置为provided,指明是在编译和**测试阶段才引入这个依赖。

总结

通过以上的总结，我们可以看到，在引入maven管理之后，整个项目的的依赖关系通过POM文件可以看得一清二楚，依赖的冲突、依赖的范围、依赖的聚合和继承可以让我们更高效的组织POM文件，让我们从jar的复制，解决jar包冲突，右键编译，运行，生成war包、部署等杂乱无章的事情中解脱出来了，从而真正的关心我们开发的模块代码该怎么写。在团队协作方面，每个人负责一个项目模块，用maven进行管理并最后组织起来不失为一种提升效率的方式，不得不说，maven是项目开发的利器。

Java设计模式之简单工厂模式

简单工厂模式又叫静态工厂方法模式，简单工厂模式由一个工厂根据传入的参数，动态决定创造哪一类产品的实例。

简单工厂模式的角色

简单工厂模式的角色分为三类
1.工厂角色:简单工厂的核心，负责创建内部实例的内部逻辑。
2.抽象产品角色: 简单工厂模式创建的所有对象的父类，描述出了所有实例中的所有公共接口。
3.具体产品角色:是简单工厂的创建目标，就是抽象产品角色的具体实现。

Java设计模式之代理设计模式

为某个对象提供一个代理，以控制对这个对象的访问。 代理类和委托类有共同的父类或父接口，这样在任何使用委托类对象的地方都可以用代理对象替代。代理类负责请求的预处理、过滤、将请求分派给委托类处理、以及委托类执行完请求后的后续处理。

代理的组成

在代理模式当中，我们需要定义三个角色：
1.抽象角色:通过接口或者抽象类声明真是角色的业务逻辑方法
2.代理角色:实现抽象角色，是真实角色的代理，通过真实角色的业务逻辑方法来实现抽象类。
3.真实角色:实现抽象角色,定义真实角色所要实现的业务逻辑，提供给代理角色调用。

代理的分为静态代理和动态代理

代理模式的使用场景

1.一个对象，一副很大的图像，需要载入很长时间
2.一个运行算，需要很长时间才能完成的计算结果，并且要在计算过程中显示运算结果
3.权限验证，在执行代理之前先验证用户的权限相关信息。
…

Java设计模式之适配器模式

适配器模式简而言之，就是将原本不能和本类兼容的对象通过转换之后换成兼容的对象，之后通过这个转换之后的对象放入本类当中进行接下来的动作。

适配器模式中的角色

1.目标接口(Target):客户所期待的接口。目标可以是具体的或抽象的类，也可以是接口。
2.需要适配的类(Adaptee):需要适配的类或适配器类。
3.适配器(Adapter):通过包装一个需要适配的对象，把原接口转换成目标接口.

Java设计模式之函数模板设计模式

模板函数设计模式，顾名思义就是在函数模板中定义一套标准，在该机制中定义好做一件事情的步骤和顺序，标准的具体实现由具体的实现机制去做。具体到面向对象当中就是设计一个抽象父类，将业务逻辑的步骤抽象成一个一个的方法，针对公共的模板方法设置为私有，针对特定的方法由子类去实现，并且由父类提供一个公有的调度方法去调度每一个步骤的顺序，子类调用父类的调度方法完成整个方法的调度。

模板函数设计模式的角色

1.抽象类:实现类模板方法，定义了算法的骨架

Java设计模式之策略设计模式

Java设计模式-策略模式

策略模式是解决某一方案的时候，方案具体的某一个实现如果有多种实现方式，那么将该实现方式定义为一个策略接口，其中的每一种实现都是一种具体的策略，这样定义之后我们可以针对需求的变化切换相应的策略即可。

策略模式的角色

1.环境类(Context):用一个ConcreteStragegy对象来配置。维护一个对Strategy对象的引用。可以定义一个接口来让Strategy访问它的数据.
2.抽象策略类(Strategy):定义所有支持算法的公共接口。Context使用这个接口来调用某个ConcreteStrategy定义的算法。
3.具体的策略类(ConcreteStrategy):以Strategy接口实现某具体算法。

策略模式的应用场景

  1. 多个类只区别在表现行为不同，可以使用Strategy模式，在运行时动态选择具体要执行的行为。
  2. 需要在不同情况下使用不同的策略(算法)，或者策略还可能在未来用其它方式来实现。
  3. 对客户隐藏具体策略(算法)的实现细节，彼此完全独立。

Java设计模式之单例设计模式

学习总结

最近在学习Java的设计模式，一直没有来得及总结所学习的东西，现在准备把所学习的设计模式一一总结。

##单例设计模式
单例设计模式是解决保证在JVM中一个类的永远只存在一个对象被外部访问，单例模式是设计模式中最简单的形式之一。这一模式的目的是使得类的一个对象成为系统中的唯一实例。要实现这一点，可以从客户端对其进行实例化开始。因此需要用一种只允许生成对象类的唯一实例的机制，“阻止”所有想要生成对象的访问。使用工厂方法来限制实例化过程。这个方法应该是静态方法（类方法），因为让类的实例去生成另一个唯一实例毫无意义。

单例模式的特点

1.单例类只能有一个实例。
2.单例类必须自己创建一个自己的唯一实例
3.单例类必须给其他对象提供着一个实例。

单例模式的类图


单例的写法有两种:懒汉模式和饿汉模式

懒汉单例模式

懒汉单例模式在多线程并发获取单例的时候会有可能会获取多个对象实例，肯能存在线程安全问题，因此需要加线程同步

Singleton通过将构造方法限定为private避免了类在外部被实例化，在同一个虚拟机范围内，Singleton的唯一实例只能通过getInstance()方法访问。
但是以上懒汉式单例的实现没有考虑线程安全问题，它是线程不安全的，并发环境下很可能出现多个Singleton实例，要实现线程安全，就需要对getInstance()方法进行改造。
1.在getInstance方法上加同步

2.双重检查锁定

3.静态内部类

相对于三种方法来说，第三种方法比前两种都好，即实现了线程安全问题，又避免了同步所带来的性能影响。

/**

  • 饿汉单例模式
    **/
    public class Singleton {
    private static Singleton instance = new Singleton();
    private Singleton() {
    }
    public static Singleton getInstance() {
    return instance;
    }
    ```
    饿汉单例模式是最简单高效的的写法，类在加载到类加载器的时候就已经创建好了一个静态对象供系统使用，以后不再改变。
